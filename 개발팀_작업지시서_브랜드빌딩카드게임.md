# 개발팀 작업 지시서 — 브랜드 빌딩 카드 게임 in 스탬프투어

> 문서 버전 v1.0 · 작성: 기획 · 대상: 서버/앱 개발팀
> 스택: **Flutter (모바일 앱)** · **Python (서버)** · **MySQL (DB)**

---

## 0. 목적과 범위

기존 ‘모바일 스탬프투어’ 앱에 신규 이벤트 모듈 **‘브랜드 빌딩 카드 게임’**을 추가한다.
관광객이 장소를 방문·인증하면 발자국 순번을 받고, 매 10번째 순번에서 ‘빨간 발자국’을 얻어 럭키드로우로 알파벳 카드를 1장 획득한다. 광고주의 브랜드명(미션 단어)을 구성하는 글자를 모두 모으면 미션을 완성하고, 선착순으로 경품을 신청한다. 부족한 글자는 사용자 간 ‘선물하기’로 주고받는다.

**이번 범위(MVP)**: 위 게임 루프 1종 + 단일/복수 캠페인 운영 + 선착순 리워드 + 선물하기 + 광고주용 기본 통계.
**범위 외**: 결제, 실물 배송 물류, 회원/인증 시스템(기존 앱 것 재사용 전제).

## 1. 시스템 아키텍처

```
[Flutter App]  ──HTTPS/JSON──▶  [Python API 서버]  ──▶  [MySQL]
   │  geolocator(위치)              │  FastAPI(권장)        │  트랜잭션/락
   │  상태관리: Riverpod(권장)        │  SQLAlchemy + Alembic │  분석 테이블
   └─ 로컬 캐시(보유 알파벳 등)        └─ 캠페인 어드민 API     └─ 읽기 전용 리포트 뷰
```

- 서버 프레임워크는 **FastAPI 권장**(비동기·자동 문서화). 팀 사정에 따라 Flask/Django 가능 — 단 아래 API 계약은 동일하게 지킬 것.
- ORM은 SQLAlchemy + 마이그레이션 Alembic 권장.
- 모든 게임 행위(스탬프·드로우·신청·선물)는 **서버가 권위(authoritative)**. 앱은 표시/요청만 한다. 클라이언트 계산을 신뢰하지 않는다.

## 2. 핵심 도메인 규칙 (가장 중요 — 모호함 없이 구현할 것)

### 2.1 발자국 순번(footprint sequence)
- 사용자가 캠페인에 속한 장소에서 스탬프를 인증하면 `stamps` 1건을 생성한다.
- 해당 사용자·캠페인 기준 **누적 순번 `sequence_no`**를 1부터 1씩 증가시켜 부여한다.
- 같은 장소 재방문 처리: 동일 장소 재인증 쿨다운(기본 24h, 캠페인 설정값)을 두어 어뷰징 방지.

### 2.2 빨간 발자국(red footprint)
- `sequence_no % red_interval == 0` 이면 그 스탬프는 `is_red = true`.
- `red_interval` 기본값 **10**(캠페인별 설정 가능).
- 빨간 발자국이 발생하면 그 즉시 **럭키드로우 1회 권한**이 생성된다(`draws` 레코드, status=`pending`).
- ⚠️ **확인 필요(2.7-A)**: ‘매 10단위’의 기준이 (a) 사용자 누적 순번인지, (b) 특정 장소의 N번째 방문자인지. 본 문서는 **(a) 사용자 누적 순번**으로 구현한다(권장·기본). 클라이언트 확정 시 반영.

### 2.3 럭키드로우(lucky draw)
- 빨간 발자국 1건당 드로우 1회. 드로우는 화면에 **카드 5장**을 제시한다.
- 5장의 후보 글자는 서버가 **캠페인 글자 풀에서 가중 추출(weighted sampling)**해 결정한다. 가중치 `draw_weight`로 특정 글자가 잘/덜 나오게 조절(= 완성률 통제 레버).
- 사용자는 5장 중 **1장을 선택**하고, 선택한 카드의 글자가 보유 인벤토리에 +1 된다. 나머지 4장은 폐기.
- 드로우 결과(제시된 5글자, 선택 위치, 획득 글자)는 `draws` / `draw_options`에 기록.
- ⚠️ **확인 필요(2.7-B)**: ‘선택’이 결과를 바꾸는 실제 선택인지, 결과는 이미 정해지고 선택은 연출인지. 본 문서는 **실제 선택(5장 중 사용자가 고른 글자를 획득)**으로 구현.

### 2.4 미션 단어 / 완성 판정
- 캠페인은 **미션 단어(mission_word)**를 가진다. 예: `IONIQ5`.
- 미션 단어를 **멀티셋(중복 포함)**으로 분해해 필요 수량을 산출한다. 예: `IONIQ5` → `I:2, O:1, N:1, Q:1, 5:1`.
- 완성 조건: 사용자의 보유 수량이 **모든 글자에서 필요 수량 이상**.
- 완성 시 ‘선물 신청하기’ 버튼 활성화. 미완성 시 비활성.
- 표시는 항상 **풀네임 슬롯**(예: `I O N I Q 5` 6칸)으로, 보유한 글자는 채워진 상태, 미보유는 빈(회색) 상태로 렌더. (브랜드 각인이 핵심 가치 — 항상 풀네임이 보이게)

### 2.5 선착순 리워드(first-come-first-served)
- 캠페인은 `reward_total`(예: 5)과 `reward_remaining`을 가진다.
- 미션 완성자가 ‘선물 신청’ 시, **원자적으로 재고를 차감**하고 선착 번호(`claim_no`)를 부여한다.
- 재고 0이면 ‘마감’ 반환. 동일 사용자 중복 신청 불가(unique 제약).
- 동시성 처리: 아래 2.6 참조.

### 2.6 동시성 · 정합성 (반드시 지킬 것)
- **선착순 재고 차감**은 다음 중 하나로 원자성 보장:
  ```sql
  UPDATE campaigns
     SET reward_remaining = reward_remaining - 1
   WHERE id = :cid AND reward_remaining > 0;
  -- affected_rows == 1 이면 성공, 0이면 마감
  ```
  성공 시 같은 트랜잭션에서 `reward_claims` insert(+ claim_no = reward_total - reward_remaining).
- **선물하기**는 단일 트랜잭션에서 보내는 쪽 수량 감소·받는 쪽 수량 증가. 보내는 쪽 수량 부족 시 롤백.
- **드로우 권한 소진**은 `draws.status` pending→done 전이로 1회성 보장(중복 사용 차단).
- 모든 쓰기 API는 **Idempotency-Key 헤더** 지원(네트워크 재시도 중복 방지).

## 3. 데이터 모델 (MySQL · DDL 초안)

> 엔진 InnoDB, 문자셋 utf8mb4. 시간은 UTC `DATETIME`. 금액/수량은 정수.

```sql
-- 광고주
CREATE TABLE advertisers (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  name          VARCHAR(120) NOT NULL,
  contact_email VARCHAR(190),
  created_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 캠페인(이벤트)
CREATE TABLE campaigns (
  id                BIGINT PRIMARY KEY AUTO_INCREMENT,
  advertiser_id     BIGINT NOT NULL,
  title             VARCHAR(150) NOT NULL,           -- 예: IONIQ 5 출시 이벤트
  mission_word      VARCHAR(40) NOT NULL,            -- 예: IONIQ5 (정규화: 대문자/공백제거)
  mission_display   VARCHAR(60) NOT NULL,            -- 화면 표기용 풀네임
  red_interval      INT NOT NULL DEFAULT 10,
  revisit_cooldown_h INT NOT NULL DEFAULT 24,
  reward_total      INT NOT NULL,
  reward_remaining  INT NOT NULL,
  status            ENUM('draft','active','paused','ended') NOT NULL DEFAULT 'draft',
  start_at          DATETIME NOT NULL,
  end_at            DATETIME NOT NULL,
  created_at        DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT fk_camp_adv FOREIGN KEY (advertiser_id) REFERENCES advertisers(id),
  INDEX idx_camp_status (status, start_at, end_at)
);

-- 캠페인 글자 풀(완성 필요 수량 + 드로우 가중치)
CREATE TABLE campaign_letters (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  campaign_id   BIGINT NOT NULL,
  letter        VARCHAR(2) NOT NULL,                 -- 'I','O','5' 등 단일 토큰
  required_count INT NOT NULL DEFAULT 1,             -- 멀티셋 기반(IONIQ5의 I=2)
  draw_weight   INT NOT NULL DEFAULT 100,            -- 가중 추출용
  UNIQUE KEY uq_camp_letter (campaign_id, letter),
  CONSTRAINT fk_cl_camp FOREIGN KEY (campaign_id) REFERENCES campaigns(id)
);

-- 스탬프 장소
CREATE TABLE places (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  campaign_id   BIGINT,                              -- NULL이면 공용 장소
  name          VARCHAR(150) NOT NULL,
  lat           DECIMAL(10,7) NOT NULL,
  lng           DECIMAL(10,7) NOT NULL,
  geofence_m    INT NOT NULL DEFAULT 100,            -- 스탬프존 반경
  image_url     VARCHAR(300),
  INDEX idx_place_camp (campaign_id)
);

-- 스탬프 인증 기록 + 발자국 순번
CREATE TABLE stamps (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id       BIGINT NOT NULL,
  campaign_id   BIGINT NOT NULL,
  place_id      BIGINT NOT NULL,
  sequence_no   INT NOT NULL,                        -- 사용자·캠페인 누적 순번
  is_red        TINYINT(1) NOT NULL DEFAULT 0,
  lat           DECIMAL(10,7), lng DECIMAL(10,7),
  created_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_stamp_user_camp (user_id, campaign_id),
  INDEX idx_stamp_revisit (user_id, place_id, created_at)
);

-- 럭키드로우(빨간 발자국 1건당 1회)
CREATE TABLE draws (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id       BIGINT NOT NULL,
  campaign_id   BIGINT NOT NULL,
  stamp_id      BIGINT NOT NULL UNIQUE,              -- 빨간 발자국 1:1
  status        ENUM('pending','done') NOT NULL DEFAULT 'pending',
  chosen_letter VARCHAR(2),
  created_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  done_at       DATETIME,
  INDEX idx_draw_user (user_id, campaign_id, status)
);

CREATE TABLE draw_options (
  id        BIGINT PRIMARY KEY AUTO_INCREMENT,
  draw_id   BIGINT NOT NULL,
  position  TINYINT NOT NULL,                        -- 0~4
  letter    VARCHAR(2) NOT NULL,
  is_chosen TINYINT(1) NOT NULL DEFAULT 0,
  UNIQUE KEY uq_draw_pos (draw_id, position),
  CONSTRAINT fk_do_draw FOREIGN KEY (draw_id) REFERENCES draws(id)
);

-- 사용자 보유 알파벳
CREATE TABLE user_letters (
  user_id     BIGINT NOT NULL,
  campaign_id BIGINT NOT NULL,
  letter      VARCHAR(2) NOT NULL,
  qty         INT NOT NULL DEFAULT 0,
  PRIMARY KEY (user_id, campaign_id, letter)
);

-- 선물하기(유저 간 카드 이전)
CREATE TABLE gifts (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  campaign_id  BIGINT NOT NULL,
  from_user_id BIGINT NOT NULL,
  to_user_id   BIGINT NOT NULL,
  letter       VARCHAR(2) NOT NULL,
  qty          INT NOT NULL DEFAULT 1,
  created_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_gift_users (from_user_id, to_user_id, campaign_id)
);

-- 선착순 리워드 신청
CREATE TABLE reward_claims (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  campaign_id BIGINT NOT NULL,
  user_id     BIGINT NOT NULL,
  claim_no    INT NOT NULL,                          -- 선착 번호(1..reward_total)
  status      ENUM('reserved','fulfilled','cancelled') NOT NULL DEFAULT 'reserved',
  created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uq_claim_user (campaign_id, user_id)
);

-- 분석 이벤트(광고주 가치)
CREATE TABLE analytics_events (
  id          BIGINT PRIMARY KEY AUTO_INCREMENT,
  campaign_id BIGINT NOT NULL,
  user_id     BIGINT,
  event_type  VARCHAR(40) NOT NULL,                  -- visit/stamp/red/draw/letter_gain/complete/claim/gift
  payload     JSON,
  created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_ae_camp_type (campaign_id, event_type, created_at)
);
```

## 4. 서버 API 명세 (REST/JSON · 인증 토큰 헤더 전제)

> 모든 게임 행위는 서버 검증. 위치 인증·재고·드로우 권한·선물 수량은 서버에서 판정.

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET  | `/v1/campaigns/{id}` | 캠페인 정보·미션 단어·풀네임·기간·잔여 경품 |
| GET  | `/v1/campaigns/{id}/me` | 내 진행 상태(보유 글자·필요 수량·완성 여부·pending 드로우) |
| POST | `/v1/campaigns/{id}/stamps` | 스탬프 인증. body: `{place_id, lat, lng}` → 순번·is_red·생성된 draw_id |
| GET  | `/v1/draws/{draw_id}` | 드로우 제시 카드 5장(position·letter는 선택 전까지 숨김 가능) |
| POST | `/v1/draws/{draw_id}/choose` | body: `{position}` → 획득 글자·갱신된 보유 현황 |
| POST | `/v1/campaigns/{id}/gifts` | body: `{to_user_id, letter, qty}` → 선물 전송(트랜잭션) |
| POST | `/v1/campaigns/{id}/claims` | 미션 완성자 선착순 신청 → `{claim_no}` 또는 `409 sold_out` |
| GET  | `/v1/admin/campaigns/{id}/stats` | 광고주용 통계(참여자·완성률·지역·응모자) |
| POST/PUT | `/v1/admin/campaigns` | 캠페인 생성/수정(미션 단어·가중치·경품수·기간) |

**스탬프 인증 응답 예시**
```json
{
  "stamp_id": 90213, "sequence_no": 10, "is_red": true,
  "draw": { "draw_id": 5521, "status": "pending" }
}
```

**드로우 선택 응답 예시**
```json
{
  "chosen_letter": "N",
  "inventory": { "I":1, "O":0, "N":1, "Q":0, "5":0 },
  "required":  { "I":2, "O":1, "N":1, "Q":1, "5":1 },
  "completed": false
}
```

**서버 검증 규칙(요약)**
- 스탬프: 캠페인 active 여부, 기간 내, geofence 반경 내(서버에서 거리 계산), 재방문 쿨다운, 위치 신뢰도(급격한 좌표 점프·모의위치 의심 시 거절/플래그).
- 드로우 선택: 본인 소유 draw·status=pending만 허용 → done 전이.
- 선물: 보내는 쪽 동일 캠페인 보유 수량 ≥ qty, 자기 자신 금지.
- 신청: 미션 완성 상태 검증 후 2.6 원자적 재고 차감.

## 5. Flutter 앱 구성

### 5.1 화면 / 라우트
| 화면 | 내용 |
|------|------|
| 홈 | 이벤트 배너, 탐색하기, 인기 장소(기존 앱 확장) |
| 이벤트 소개 | 신청 방법(빨간 발자국→카드→완성→신청) |
| 지도/탐색 | 장소 목록·거리·스탬프존 표시 |
| 스탬프 인증 | 위치 확인 → 인증 요청 → 결과 |
| 빨간 발자국 모달 | “빨간 발자국 획득” → ‘카드 선택하러 가기’ |
| 카드 선택(드로우) 모달 | 5장 제시 → 1장 선택 → 글자 공개 |
| 보관함/미션 진행 | **풀네임 슬롯**(채움/빈칸) + 보유 수량 + 완성 시 신청 버튼 |
| 선물하기 | 친구 선택·글자 선택·전송 |
| 선물 신청 | 선착순·잔여 수량·신청 결과 |

### 5.2 상태관리·패키지(권장)
- 상태관리 **Riverpod**(or Bloc). 위치 `geolocator` + 권한 `permission_handler`.
- 네트워크 `dio`(+Idempotency-Key 인터셉터), 모델 `freezed`/`json_serializable`.
- 로컬 캐시: 보유 글자·진행 상태(서버 응답 미러). **단, 완성/재고 판정은 항상 서버 응답 신뢰.**

### 5.3 핵심 UX 규칙
- 미션 슬롯은 어느 화면에서든 **항상 풀네임 전체**를 보여준다(각인 목적). 빈 칸은 회색, 채운 칸은 강조.
- 드로우는 기대감(카드 뒷면→선택→공개) 연출. 결과는 서버값으로 표시.
- 네트워크 실패 시 재시도(동일 Idempotency-Key)로 중복 인증/중복 신청 방지.
- 신청 마감/중복 신청/완성 미달 등은 명확한 에러 메시지.

## 6. 보안 · 어뷰징 대응
- 위치 위조 방지: 서버 측 거리 검증 + 모의위치/좌표 점프 탐지 + 단시간 다중 인증 rate limit.
- 모든 쓰기 API 인증 필수, 본인 리소스만 접근.
- 선물/신청 트랜잭션 격리, 재고 음수 불가(DB 제약 + 원자적 UPDATE).
- 개인정보(응모자) 최소 수집·동의 기반·암호화 보관(법무 확인 항목 F).

## 7. 분석 / 광고주 가치 지표
`analytics_events`로 퍼널 적재: 방문→스탬프→빨간발자국→드로우→글자획득→완성→신청→선물.
광고주 대시보드 핵심 지표: 참여자 수(UU), 미션 완성률, 평균 완성 소요일, 지역 분포, 선물하기 발생량(바이럴), 응모자 수(리드). → 별도 어드민 API/리포트 뷰로 제공.

## 8. 마일스톤 / 작업 분해(WBS)
| 단계 | 산출물 | 완료 기준(DoD) |
|------|--------|----------------|

## 9. 인수 기준(전체)
- 한 사용자가 방문→완성→선착순 신청까지 서버 검증 하에 정상 동작.
- 선착순 재고가 경합 상황에서도 절대 초과 지급되지 않음.
- 미션 슬롯이 모든 화면에서 풀네임으로 정확히(철자 일치) 렌더됨.
- 2.7 표의 확인 필요 항목은 착수 전 기획/클라이언트와 확정.
